# 知识库问答、引用展示与反馈闭环需求梳理

任务 ID：976a223c-3a37-4b4d-a747-97ba17470b6b  
角色：产品经理  
范围：只读分析与产品文档化，不直接修改业务代码  
分析对象：`internal-ai-assistant`

---

## 1. 背景与业务目标

用户目标是把当前内部 AI 问答系统升级为可信的知识库问答助手：

1. 每次 AI 回答都必须基于用户有权限访问的知识库内容。
2. 每次 AI 回复的聊天框下方必须展示本次引用的知识库文件。
3. 用户点击引用文件后可以打开查看原文、预览或命中片段。
4. AI 回答完成后提供反馈入口，用户填写反馈后提交给管理员。
5. 优化前端聊天体验，并给出项目整体升级建议。

产品目标：降低 AI 幻觉风险，提高回答可追溯性，让用户和管理员形成“提问—引用核查—反馈—知识库优化”的闭环。

## 2. 当前项目现状（只读分析结论）

### 2.1 技术与功能现状

项目主体为 `internal-ai-assistant`：

- 后端：FastAPI、SQLAlchemy、SQLite、Qdrant 可选向量检索。
- 前端：Vue 3 + Element Plus；同时后端 `main.py` 内也包含一套内嵌 HTML 页面。
- 已有核心数据表：用户、岗位组、文档、文档分片、文档处理状态、聊天会话、聊天消息、审计日志、后台任务、设置。
- 已支持：
  - 登录与管理员权限。
  - 管理员上传知识库文件。
  - PDF、Word、Excel、CSV、TXT、Markdown、图片附件解析或 OCR。
  - 文档分片和向量检索。
  - 文档按岗位组授权。
  - 普通用户只能检索授权文档和本人聊天附件。
  - `/api/chat` 返回 `answer` 与 `sources`。

### 2.2 已有能力与用户目标的匹配度

| 用户目标 | 当前状态 | 差距 |
|---|---|---|
| AI 基于知识库回答 | 已有检索和提示词约束 | 仍需产品级强约束：无引用不得事实性回答；需可验证引用 |
| 回复下方展示引用文件 | Vue 页面当前是全局 sources；内嵌页是右侧抽屉 | 未做到“每条 AI 回复下方独立引用” |
| 点击引用打开查看 | 管理端可看文档 chunks | 用户侧未定义点击引用查看原文/片段入口 |
| 回答反馈给管理员 | 未发现反馈表、接口或管理页 | 需要新增完整反馈闭环 |
| 前端优化 | 内嵌页体验较完整；Vue 页较简陋 | 需要统一前端入口和交互模式 |

### 2.3 关键缺口

1. `sources` 未与具体 assistant 消息持久绑定，历史会话无法恢复每条回答对应引用。
2. 没有反馈数据模型、反馈提交接口、管理员反馈管理界面。
3. 用户侧没有“点击引用打开文件/片段”的明确交互和权限校验方案。
4. 当前 README 与部分源码中文内容存在编码显示异常风险。
5. 当前存在 Vue 前端与后端内嵌页面两套界面，产品体验和后续维护方向需统一。
6. 未发现自动化测试、Lint 或端到端验收脚本。

## 4. 核心流程

### 4.1 知识库问答流程

1. 用户登录。
2. 用户输入问题。
3. 系统显示“正在检索授权知识库”。
4. 后端根据用户身份和岗位组检索：
   - 管理员可检索管理范围内知识库；
   - 普通员工只能检索授权岗位组文档；
   - 个人聊天附件只能本人检索。
5. 检索结果为空：
   - 返回“未在授权知识库中找到依据”的兜底回答；
   - 不允许模型自由编造事实。
6. 检索结果非空：
   - 模型只基于检索片段生成回答；
   - 回答中或回答下方必须包含引用来源。
7. 系统保存用户消息、AI 消息、引用快照。
8. 前端在该条 AI 回复下方展示引用列表和反馈按钮。

### 4.2 引用查看流程

1. 用户点击某条 AI 回复下方的引用卡片。
2. 系统再次校验用户是否仍有该文档/片段访问权限。
3. 有权限：
   - PDF：优先打开文件预览并定位页码；初版可打开预览并显示页码提示。
   - TXT/Markdown/CSV/Excel/Word：初版可打开解析后的文本片段详情；后续增强原文件预览。
   - 图片：打开图片预览或 OCR 文本。
4. 无权限、文件不存在、解析失败：显示明确错误提示。

### 4.3 回答反馈流程

1. AI 回答展示完成。
2. 回答下方显示反馈入口：有帮助、不准确、引用不对、内容缺失、其他。
3. 用户点击后弹出反馈输入框。
4. 用户提交反馈。
5. 后端记录反馈、关联用户、会话、消息、问题、回答、引用快照。
6. 管理员在后台查看反馈列表，筛选、处理、备注。

## 5. 用户故事、优先级与验收标准

### US-01：AI 每次回答必须基于授权知识库内容

优先级：P0  
角色：普通员工  
故事：作为员工，我希望 AI 只根据我有权限访问的知识库回答，避免看到编造或未授权内容。

验收标准：

- 每次 `/api/chat` 请求必须先执行知识库检索。
- 检索范围必须经过用户权限过滤。
- 无命中内容时，回答必须明确说明“未在授权知识库中找到依据”，不得生成事实性答案。
- 有命中内容时，模型提示词必须要求只基于命中片段回答。
- 回答结果必须返回引用来源；无引用的事实性回答视为失败。
- 管理员、普通员工、无岗位组员工的检索边界均需被验证。

边界条件：

- 用户问候、系统能力咨询等非知识库问题，可以回答系统能力，但不得编造业务事实。
- 检索命中低相关内容时，应提示“依据不足”，而不是强行回答。
- 向量库不可用时可回退 SQLite，但权限规则必须保持一致。

---

### US-02：每条 AI 回复下方展示引用文件

优先级：P0  
角色：普通员工  
故事：作为员工，我希望每条 AI 回复下面都能看到本次回答引用了哪些知识库文件。

验收标准：

- 每条 assistant 消息下方独立展示引用列表，而不是只显示最近一次全局引用。
- 引用列表至少展示：
  - 文档标题；
  - 文件名；
  - 页码或片段序号；
  - 来源类型：知识库文档 / 个人附件；
  - 命中文本摘要；
  - 查看按钮。
- 历史会话重新打开后，引用仍跟随原 AI 回复展示。
- 同一回答多个引用时按相关性排序。
- 无引用时展示“本次未命中可引用内容”，并配合无依据兜底回答。

建议数据要求：

- 新增或扩展消息引用快照数据，例如：`chat_message_sources`。
- 保存字段建议：`message_id`、`document_id`、`document_title`、`filename`、`page_number`、`chunk_id`、`chunk_index`、`content_preview`、`score`、`source_type`、`created_at`。

---

### US-03：点击引用可打开查看

优先级：P0  
角色：普通员工  
故事：作为员工，我希望点击引用后可以查看原始文件或命中片段，确认 AI 回答依据。

验收标准：

- 引用卡片或“查看”按钮可点击。
- 点击后必须重新校验当前用户权限。
- 有权限时展示可读内容：
  - PDF：预览文件，尽可能定位页码；
  - Word/Excel/CSV/TXT/Markdown：展示解析文本和命中片段；
  - 图片：展示图片或 OCR 结果。
- 无权限时返回 403，并提示“无权查看该引用”。
- 文件被删除或丢失时返回 404，并提示“文件不存在或已被管理员删除”。
- 打开引用不应暴露服务器真实文件路径。

初版建议：

- 先实现“片段详情弹窗/抽屉”，确保所有文件类型都有一致体验。
- PDF 原文定位、Office 在线预览作为 P1/P2 增强。

### US-04：AI 回答完成后提供反馈按钮

优先级：P0  
角色：普通员工  
故事：作为员工，我希望可以对 AI 的某条回答提交反馈，让管理员知道问题所在。

验收标准：

- 每条 AI 回复生成完成后，下方显示反馈入口。
- 反馈入口至少支持：
  - 有帮助；
  - 不准确；
  - 引用不对；
  - 没有解决问题；
  - 其他。
- 用户可填写补充说明。
- 提交后显示成功状态，例如“已反馈”。
- 反馈必须关联具体 assistant message，而不是只关联会话。
- 未登录用户不能提交反馈。
- 反馈提交失败时保留用户输入并提示原因。

建议数据字段：

- `id`
- `user_id`
- `username_snapshot`
- `session_id`
- `message_id`
- `question_snapshot`
- `answer_snapshot`
- `sources_snapshot_json`
- `feedback_type`
- `feedback_text`
- `status`：pending / reviewing / resolved / ignored
- `admin_note`
- `created_at`
- `updated_at`

## 7. 非功能需求

### 7.1 权限与安全

- 所有引用查看接口必须做服务端权限校验。
- 反馈详情仅管理员可见；普通用户只可提交和查看自己反馈状态（如需要）。
- 不暴露服务器本地路径。
- 审计关键操作：文档权限变更、反馈处理、文档删除、重解析。

### 7.2 可追溯性

- AI 回答、引用快照、反馈必须可关联。
- 即使文档后续被修改或删除，历史回答仍应保留当时的引用摘要和反馈记录。

### 7.3 可用性

- Qdrant 不可用时允许回退 SQLite，但需要明确日志和管理端状态提示。
- 文档解析异步化时，前端需提示“解析中，完成后可参与检索”。

### 7.4 兼容性

- 支持桌面端和移动端浏览器。
- 初版文件查看可采用统一片段详情，不强依赖复杂 Office 预览。

## 8. 风险与边界条件

### 8.1 模型幻觉风险

风险：模型可能根据常识补充知识库之外内容。  
应对：无引用不事实回答；提示词约束；前端始终显示引用；后端可增加“必须至少 N 条引用才生成业务答案”的策略。

### 8.2 检索质量风险

风险：local-hash embedding 召回质量有限，可能命中不相关片段。  
应对：引入相关性阈值；支持更高质量 embedding；管理员可通过反馈发现低质量问答。

### 8.3 权限泄露风险

风险：引用 document_id 被猜测后越权查看。  
应对：引用详情接口必须重新计算权限，不信任前端传参。

### 8.4 历史引用一致性风险

风险：只保存 document_id，文档后续变更导致历史回答依据变化。  
应对：保存引用快照，包括 title、filename、page、chunk content preview、score。

### 8.5 文件预览复杂度风险

风险：PDF、Word、Excel、图片预览技术路径不同。  
应对：P0 先统一展示解析片段；P1/P2 再做原文预览增强。

### 8.6 前端双入口风险

风险：Vue 前端和后端内嵌 HTML 同时存在，需求落地可能重复开发。  
应对：团队需先决定主前端入口；建议以 Vue 前端为长期维护方向，后端内嵌页仅保留调试或逐步下线。

### 8.7 编码与质量风险

风险：现有 README 和部分中文文本显示乱码；缺少测试和构建校验。  
应对：统一 UTF-8；补充前后端测试；上线前执行冒烟测试。

## 10. 建议实施拆分

### 10.1 第一阶段：可信回答与引用展示（P0）

- 后端保存 assistant message id。
- 新增消息引用快照表。
- `/api/chat` 返回 message_id 和 sources。
- 历史会话接口返回每条 assistant 消息的 sources。
- 前端在每条 AI 回复下展示引用卡片。
- 无引用时强制兜底回答。

### 10.2 第二阶段：引用查看（P0）

- 新增引用详情/片段查看接口。
- 服务端做权限校验。
- 前端实现引用详情弹窗或抽屉。
- 初版所有文件类型统一查看解析片段。

### 10.3 第三阶段：反馈闭环（P0/P1）

- 新增反馈表和提交接口。
- 前端每条 AI 回复下提供反馈入口。
- 管理端新增反馈管理页。
- 支持状态流转和管理员备注。

### 10.4 第四阶段：体验与质量升级（P1/P2）

- 统一前端入口。
- 优化移动端体验。
- 增加 PDF 页码定位和 Office 预览。
- 增加测试、Lint、构建检查。
- 增加知识库质量看板。
