# AI 修改前必读

> 目的：防止 AI 在长对话、上下文压缩或跨天继续时误改项目、误杀进程、误判业务规则。

## 每次开始修改前必须做

1. 先阅读以下三份文档：
   - `docs/AI修改前必读.md`
   - `docs/业务规则回归清单.md`
   - `docs/AI修改记录.md`
2. 再确认当前工作区状态：
   - `git status --short`
   - 如涉及回退/提交，先查看最近提交：`git log --oneline -5`
3. 先说明计划，再执行修改。
4. 用户的最新明确指令优先级最高；本文档是默认规则，不是永久不可改规则。

## 用户指令优先级

- 一切以用户最新输出为准。
- 如果用户明确改变规则、端口、流程、提交目标，应更新本文档或修改记录。
- 如果用户要求“直接执行”，仍需遵守安全边界：不碰无关项目、不删除不确定文件、不回退未确认改动。

## 当前项目固定信息

- 项目目录：`d:\ai\speceappdate\知识库\internal-ai-assistant`
- 后端端口：`8000`
- 前端端口：`5174`
- `5173` 是其他项目/程序端口，不允许停止、占用、重启或作为本项目前端端口。
- 前端 Vite 配置应保持 `5174`。
- 目标 GitHub 仓库：`https://github.com/liaaa00/zhishiku.git`

## 进程操作规则

- 不允许操作 `5173`。
- 只允许操作本项目相关进程：
  - 后端 `8000`
  - 前端 `5174`
- 重启后端前必须说明原因。只有后端代码变化需要生效时才重启 `8000`。
- 修改前端代码通常只需确认 Vite/HMR 或重启 `5174`，不要动后端。
- 如果发现多个 uvicorn / vite 残留，只清理能确认属于 `internal-ai-assistant` 的进程。

## 文件修改规则

- 所有文本文件写入/编辑优先使用 SpectrAI 文件工具：
  - `spectrai_write_file`
  - `spectrai_edit_file`
  - `spectrai_create_file`
  - `spectrai_delete_file`
- 禁止用 shell 重定向、`echo >`、`sed -i`、Python/Node 内联脚本直接写中文文件，避免编码破坏。
- 读取文件可以用 shell/rg/Get-Content。
- Windows 搜索文件内容优先使用 `rg`。

## Git 与提交规则

- 提交前必须先说明：本次提交包含哪些改动、哪些测试已通过。
- 当前目标远程仓库：`https://github.com/liaaa00/zhishiku.git`
- 如果 remote 未配置，可建议配置：
  - `git remote add origin https://github.com/liaaa00/zhishiku.git`
- 只有用户明确要求提交/推送时才执行 commit/push。
- 回退优先使用安全回退：`git revert <commit>`。
- 除非用户明确要求，不使用 `git reset --hard`。

## PageIndex / RAG 注意事项

- 用户希望尽量全部文件都走 PageIndex。
- 回答优先来自 PageIndex 结构树；普通向量检索可作为补充。
- 扫描版 PDF 走 PDF OCR / 视觉解析。
- 编辑片段后应提示“请重建高级索引”。
- PageIndex 的 `chunk_index` 可能是字符串，例如 `pageindex:0000`，不要强制 `int()`。

## 聊天路由注意事项

- 普通对话、助手身份问题不应调用知识库：
  - `你是谁`
  - `你可以做什么`
  - `你好`
- 新窗口无上下文的编辑/整理类请求应直接交给普通模型回答或追问，不应检索知识库：
  - `把内容整理成表格`
  - `帮我改写一下`
  - `翻译成英文`
- 明确指向知识库/文档/资料/附件/公司数据的问题才检索：
  - `总结我现在可读的文档`
  - `把文档内容整理成表格`
  - `入职工单数据整理成表格`

## 已知易错点

- `Failed to fetch` 常见原因是前端 `5174` 停止，而不是模型不回答。
- 后端空闲不会自动停止；如果停止，多半是崩溃、窗口关闭、重启、或旧 reload 进程残留。
- 不要把浏览器窗口还在误认为 Vite 服务还在，要检查 `5174` 是否 Listen。
- 旧后端 `--reload` 可能残留多个父子进程，必要时只清理本项目 uvicorn 进程。
