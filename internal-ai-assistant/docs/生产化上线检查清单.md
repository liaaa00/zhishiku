# 生产化上线检查清单

本文用于把当前 `internal-ai-assistant` 从“能跑通”推进到“可实际投入内部试运行”。检查项按必须完成、建议完成、上线后持续执行分层。

## 1. 当前系统形态

- 前端：Vue SPA，经 Nginx 对外提供 `/chat`、`/admin`，默认端口 `8080`。
- 后端：FastAPI，容器内部端口 `8000`，提供认证、文档管理、问答、反馈、评测、权限、图谱、表格规则查询等 API。
- 向量库：Qdrant，默认集合 `internal_ai_chunks`。
- 主数据：SQLite `backend/data/app.db`。
- 原始文件：`backend/data/uploads/`。
- 高级结构索引：`backend/data/pageindex/`。
- 表格精确问答：优先走确定性表格查询；涉及分析/建议的问题走“表格事实块 + LLM 分析”。
- 回归基线：单元测试 + 真实检索评测 `24/24`，其中检索评测只验路由与证据命中，不验最终回答文字完全一致。

## 2. 上线前必须完成

### 2.1 配置安全

- [ ] `.env` 存在且不提交到 Git。
- [ ] `APP_ENV=production`。
- [ ] `JWT_SECRET` 已替换为高强度随机值，不能是 `please-change-this-secret`。
- [ ] `DEFAULT_ADMIN_PASSWORD` 已替换为强口令，不能是 `change-this-admin-password`。
- [ ] `DEFAULT_ADMIN_USERNAME` 明确设置。
- [ ] `OPENAI_API_KEY` 已配置为正式可用 Key。
- [ ] `OPENAI_BASE_URL` 与 `CHAT_MODEL` 与实际供应商一致，例如 DeepSeek/OpenAI-compatible。
- [ ] `CORS_ORIGINS` 设置为真实访问域名，不保留无关 localhost。
- [ ] `MAX_UPLOAD_MB` 与 Nginx `client_max_body_size` 保持一致或前者更小。

### 2.2 Embedding 与向量库

- [ ] 生产环境不能使用 `EMBEDDING_PROVIDER=local` + `EMBEDDING_MODEL=local-hash`。
- [ ] 生产环境应使用 `EMBEDDING_PROVIDER=openai-compatible` 或等价远程 embedding。
- [ ] `EMBEDDING_MODEL`、`EMBEDDING_API_KEY`、`EMBEDDING_BASE_URL` 配置完成。
- [ ] 确认 embedding 维度与 Qdrant 集合一致；换模型后必须重建向量。
- [ ] `VECTOR_BACKEND=qdrant`。
- [ ] `QDRANT_URL=http://qdrant:6333`（Docker Compose 内）或实际 Qdrant 地址。
- [ ] `QDRANT_COLLECTION` 固定，不随意改名；改名等同新建索引。

### 2.3 持久化与备份

必须纳入备份：

- [ ] `backend/data/app.db`：用户、权限、文档元数据、分块、表格行、反馈、审计、评测配置、设置项。
- [ ] `backend/data/uploads/`：原始文档；没有它将无法重新解析、查看来源文件。
- [ ] `backend/data/pageindex/`：高级结构索引缓存；丢失后可重建，但会影响恢复时间。
- [ ] Docker volume `qdrant_storage`：Qdrant 向量库；丢失后可从 SQLite 分块重建，但耗时且依赖 embedding 服务。
- [ ] `.env`：只进入安全密码库/服务器密钥管理，不进入普通文档仓库。

备份策略建议：

- [ ] 每日一次增量/快照备份，保留至少 14 天。
- [ ] 每周一次离线归档，保留至少 8 周。
- [ ] 每次批量导入真实知识库前，先做一次手动快照。
- [ ] 每月至少演练一次恢复：恢复到临时环境，验证 `/api/health`、后台文档列表、3 个核心问题。

### 2.4 权限与账号

- [ ] 首个管理员账号登录后立即修改初始密码。
- [ ] 普通用户注册后默认为 `pending`，必须管理员审批后启用。
- [ ] 建立用户组，例如：总部、北仑、财务、人事、测试。
- [ ] 文档上传后必须设置可见用户组；敏感资料不能留空泛授权。
- [ ] 用不同用户实测：有权限能问到，无权限看不到来源、打不开原文。
- [ ] 审计日志能记录登录、上传、权限变更、向量重建、评测等关键操作。

### 2.5 真实知识库导入

- [ ] 第一批只导入 10-30 份高频、高价值、低争议文档。
- [ ] 每份文档确认：标题、文档类型、知识范围、可见用户组。
- [ ] 上传后后台状态必须达到 `ready/searchable`。
- [ ] 表格类资料必须抽出结构化行，关键列语义要经过业务确认。
- [ ] 扫描件 PDF 或图片类资料必须抽样验证 OCR 质量。
- [ ] 低置信分类 `needs_review` 必须人工确认。

### 2.6 评测门禁

上线前至少执行：

```powershell
cd internal-ai-assistant\backend
python -X utf8 -m pytest tests/unit -q
python -X utf8 tests/qa_retrieval_eval_runner.py --real-db --cases tests/retrieval_eval_real_cases.json
```

通过标准：

- [ ] 单元测试通过。
- [ ] 真实检索评测保持 `24/24` 或新基线经过人工确认。
- [ ] 分公司统计口径保持 36：名称栏有效即计入，宁波北仑总部条目计入。
- [ ] 对无证据问题能拒答或提示资料不足，不把弱相关材料当依据。

## 3. 建议完成后再扩大试用

- [ ] 配置反向代理 HTTPS 和正式域名。
- [ ] Nginx/网关限制只开放前端入口，后端 `8000` 不直接暴露公网。
- [ ] Qdrant `6333/6334` 不对普通用户网络开放。
- [ ] 建立管理员操作 SOP：上传、重建、权限、评测、回滚。
- [ ] 增加“用户反馈 -> 评测用例 -> 回归”的闭环。
- [ ] 明确资料负责人：每个知识库领域至少有一个业务确认人。
- [ ] 建立大模型/embedding Key 的额度、告警和停用流程。

## 4. 首批试运行验收问题

必须覆盖以下类型：

1. 标准流程问答：员工怎么签署电子劳动合同？
2. 口语化问答：我收到签合同通知了，从哪里进去处理？
3. 角色冲突：合同组在工单系统里怎么处理劳动合同续签交付？
4. 表格精确查询：2026年3月北仑派单截止时间是什么时候？
5. 表格统计：浙江目前开了多少个城市的分公司，分别都叫什么名字？
6. 表格分析：分析一下北仑分公司开设进度有什么风险和下一步建议。
7. 无证据防幻觉：深圳2027年医保缴费规则是什么？
8. 权限隔离：无权限用户询问敏感资料时不能返回内容或来源。

## 5. 不满足时不能上线的红线

- 仍使用默认管理员密码或默认 JWT 密钥。
- 生产环境仍使用 `local-hash` embedding。
- 没有任何数据备份和恢复演练。
- 管理后台可被未授权用户访问。
- 上传文档后大批量处于 failed/不可检索状态。
- 回归评测明显下降且没有业务确认的新基线。
- 表格统计口径与用户显式确认冲突。
