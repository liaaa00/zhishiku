# 业务评测集建设方案

本文定义如何把真实业务问题沉淀为可回归的评测集，保证系统上线后不是“凭感觉好用”，而是能持续验证检索、路由、权限和表格口径。

## 1. 评测边界

当前真实检索评测的核心目标：

- 验证是否走对检索路由。
- 验证是否命中正确证据。
- 验证是否避开明显错误资料。
- 验证无证据时不强答。

重要约束：

- 检索评测不要求最终回答文字完全一致。
- 表格确定性问题要验证统计数、关键字段和来源。
- 用户显式确认的业务口径优先于启发式推断。

## 2. 用例分类

每个业务域至少覆盖以下类型：

| 类型 | 目的 | 示例 |
|---|---|---|
| 高频标准问法 | 验证常规命中 | 员工怎么签署电子劳动合同？ |
| 口语化问法 | 验证语义泛化 | 我收到签合同通知了，从哪里进去处理？ |
| 角色冲突 | 防止相似文档误召回 | 合同组在工单系统里怎么处理续签交付？ |
| 表格精确查询 | 验证表格路由和字段 | 2026年3月北仑派单截止时间是什么时候？ |
| 表格统计 | 验证统计口径 | 浙江目前开了多少个城市的分公司？ |
| 表格分析 | 验证事实块 + LLM 分析 | 分析北仑分公司开设进度风险 |
| 无证据防幻觉 | 验证拒答 | 深圳2027年医保缴费规则是什么？ |
| 权限隔离 | 验证访问控制 | 普通用户询问敏感财务制度 |
| 时效/版本 | 验证最新资料优先 | 2026年最新规则是什么？ |

## 3. 用例字段规范

评测文件位置：

```text
backend/tests/retrieval_eval_real_cases.json
```

推荐字段：

```json
{
  "id": "real_domain_topic_001",
  "category": "高频标准问法",
  "question": "用户实际会问的问题",
  "why": "为什么这个问题重要，以及应该命中什么",
  "expected": {
    "backend": "table",
    "top_n": 5,
    "top_n_must_match_titles": ["应该命中的标题关键词"],
    "top_n_must_not_match_titles": ["不应命中的标题关键词"],
    "must_include_terms": ["证据中必须出现的词"],
    "must_have_positive_signals": ["正向信号"],
    "must_match_contexts": [["同一片段中应共同出现的词A", "词B"]],
    "max_contexts": 0
  }
}
```

说明：

- `backend=table` 用于明确应该走表格/结构化路径的题。
- `top_n_must_match_titles` 用于检查前 N 条来源是否命中正确标题。
- `top_n_must_not_match_titles` 用于防止被相似但错误的资料压过。
- `must_include_terms` 用于检查证据内容。
- `max_contexts=0` 常用于无证据问题。

## 4. 新增用例流程

1. 从真实用户问题、后台反馈、业务验收中收集问题。
2. 业务负责人确认“正确证据”和“错误证据”。
3. 添加到 `retrieval_eval_real_cases.json`。
4. 先在当前库运行，确认测试本身不是错的。
5. 如果用例失败，判断是系统问题还是资料缺失。
6. 修复后运行全量评测。
7. 将新增口径写入相关文档。

## 5. 分公司统计专项基线

该专项用于防止表格统计被回归破坏。

### 5.1 确认口径

- 统计字段：`当前进度-4.开设公司名称`，标准别名 `branch_company`。
- 城市字段：`城市`，标准别名 `city`。
- 有效公司名：非空，且不是 `未开设/无/暂无/否`。
- 不要求包含“分公司”。
- 不额外要求银行、社保完成，除非用户显式问“银行和社保都办好”。
- 宁波北仑总部名称计入。
- 总数：36。

### 5.2 必测问题

```text
浙江目前开了多少个城市的分公司，分别都叫什么名字？
```

期望：

- 返回 36。
- 明细包含 `宁波北仑`。
- 明细包含 `无锡`。
- 明细展示城市 + 开设公司名称，而不是只有城市名。

```text
银行和社保都办好的城市有哪些？
```

期望：

- 只有用户显式问银行/社保完成时，才附加这些筛选。

```text
分析一下北仑分公司开设进度有什么风险和下一步建议。
```

期望：

- 走表格事实块 + LLM 分析。
- 不改写确定性字段。
- 表格没有提供的信息必须说“表格未提供”。

## 6. 发布门禁

每次上线前：

```powershell
cd internal-ai-assistant\backend
python -X utf8 -m pytest tests/unit -q
python -X utf8 tests/qa_retrieval_eval_runner.py --real-db --cases tests/retrieval_eval_real_cases.json
```

若失败：

1. 不直接改期望值。
2. 先看召回来源是否合理。
3. 若业务口径变化，必须有业务确认记录。
4. 修复后补单测或评测用例。
5. 更新 `docs/业务规则回归清单.md` 或专项口径文档。

## 7. 指标建议

试运行初期建议每周统计：

- 真实问题总数。
- 有帮助/不够好反馈比例。
- 无来源回答比例。
- 表格问题命中率。
- 无证据拒答正确率。
- 权限拦截正确率。
- 新增评测用例数量。
- 回归评测通过率。

目标不是追求一次性完美，而是形成闭环：

```text
用户问题/反馈 → 业务确认 → 加入评测 → 修复/补资料 → 全量回归 → 上线
```
