# 真实知识库导入与验收流程

本文用于规范真实资料进入系统的步骤，避免“资料导入了但不可检索、权限错、口径错、无法回滚”。

## 1. 导入原则

1. 先少量高价值资料试运行，再扩大范围。
2. 每批资料都要有业务负责人确认口径。
3. 正式资料和测试资料分开：正式用 `production`，实验/临时用 `test`。
4. 表格类资料必须确认列语义和统计口径，不能只看上传成功。
5. 权限先设计再导入，不能默认所有人可见敏感资料。
6. 导入前先备份，导入后做评测。

## 2. 批次准备

每批导入前建立一张导入登记表，至少包含：

| 字段 | 说明 |
|---|---|
| 文件名 | 原始文件名 |
| 资料标题 | 页面显示标题，必要时重命名 |
| 业务域 | 人事、合同、工单、社保、财务等 |
| 文档类型 | table / employee_guide / workorder / form / policy / general |
| 知识范围 | production 或 test |
| 权限用户组 | 哪些用户组可见 |
| 业务负责人 | 谁确认内容和口径 |
| 验收问题 | 至少 1-3 个该文档应该回答的问题 |
| 禁答问题 | 没有资料时必须说不足的问题 |

## 3. 导入步骤

### 3.1 导入前

- [ ] 已备份 `backend/data` 和 Qdrant volume。
- [ ] `.env` 已通过 `python -X utf8 scripts\preflight_check.py`。
- [ ] 管理员账号可登录。
- [ ] 用户组已经建好。
- [ ] 本批资料已完成脱敏检查。

### 3.2 上传

在 `/admin` 上传资料。系统支持：

- PDF
- Word `.docx`
- PowerPoint `.pptx`
- Excel `.xlsx`
- CSV
- TXT / Markdown
- 图片 PNG/JPG/JPEG/WEBP/GIF（走 OCR）

旧版 `.doc/.ppt/.xls` 需先另存为新格式。

上传时设置：

- `knowledge_scope=production`：正式知识库。
- `knowledge_scope=test`：试验资料，不应影响正式问答。
- `document_kind=auto`：让系统初判；低置信时必须人工确认。

### 3.3 等待处理完成

后台处理链路：

```text
保存原文件
  ↓
写入 documents / processing status
  ↓
解析文本、表格或 OCR
  ↓
生成 document_chunks / document_table_rows
  ↓
写入 embedding 与 Qdrant
  ↓
构建 PageIndex 结构索引
  ↓
触发知识图谱抽取
  ↓
状态 ready/searchable
```

验收标准：

- [ ] 文档状态为 `ready`。
- [ ] `searchable=true`。
- [ ] chunks 数量大于 0。
- [ ] 表格文档有结构化行。
- [ ] PageIndex 不阻塞基础检索；若失败，记录是否需要重建。
- [ ] 图谱抽取失败不应阻塞基础问答，但应记录后续处理。

## 4. 文档质量验收

### 4.1 普通文档

每份文档至少问：

1. 一个标准问法。
2. 一个口语化问法。
3. 一个相似但不应命中的干扰问法。

通过标准：

- 前 3-5 条来源应包含该文档或正确资料。
- 答案必须引用来源。
- 没有资料时不能编造。

### 4.2 表格文档

表格资料必须额外确认：

- [ ] 关键列名是否识别正确。
- [ ] 列别名是否需要人工确认。
- [ ] 统计口径是否来自用户显式声明，而不是启发式猜测。
- [ ] 城市、公司名称、日期、状态、金额等字段是否可被筛选。
- [ ] 统计问题和明细问题都能回答。
- [ ] 涉及分析/建议时，系统走“表格事实块 + LLM 分析”，不能改写确定性统计数。

当前已确认的分公司统计口径：

- 问题：`浙江目前开了多少个城市的分公司，分别都叫什么名字？`
- 口径：`当前进度-4.开设公司名称` / `branch_company` 有有效名称即计入。
- 有效：非空，且不是 `未开设/无/暂无/否`。
- 不要求名称包含“分公司”。
- 宁波北仑 `外服（浙江）企业服务有限公司` 计入。
- 期望数量：36。
- 明细应同时展示城市和开设公司名称。

### 4.3 权限验收

对每批文档至少创建两个测试用户：

- 有权限用户。
- 无权限用户。

验证：

- [ ] 有权限用户能检索到文档内容。
- [ ] 无权限用户不能从问答、来源、原文链接看到该文档。
- [ ] 管理员能在后台看到并调整权限。
- [ ] 权限变更后，必要时重建向量，使 Qdrant payload 同步。

## 5. 导入后评测

### 5.1 把验收问题加入评测集

每批至少新增：

- 高频标准问法。
- 口语化问法。
- 角色/权限冲突问法。
- 无证据防幻觉问法。
- 表格精确查询/统计问法（如适用）。

评测文件：

```text
backend/tests/retrieval_eval_real_cases.json
```

### 5.2 运行评测

```powershell
cd internal-ai-assistant\backend
python -X utf8 tests/qa_retrieval_eval_runner.py --real-db --cases tests/retrieval_eval_real_cases.json
```

通过标准：

- 新增用例通过。
- 原有 24/24 不下降；如业务资料变化导致基线变化，必须写明原因并经业务负责人确认。

## 6. 问题处理

| 问题 | 处理 |
|---|---|
| 没有解析出文本 | 检查文件是否扫描件，必要时走 OCR 或换清晰文件 |
| 表格列识别错 | 在表格 schema/alias 中确认列别名，补测试 |
| 搜不到文档 | 检查 knowledge_scope、权限、chunks、Qdrant 状态 |
| 答案编造 | 增加无证据评测，调整提示词/检索阈值 |
| 统计口径错 | 以用户显式口径优先，更新规则、文档和单测 |
| 权限泄漏 | 立即下线该文档或收紧用户组，重建向量并审计日志 |
